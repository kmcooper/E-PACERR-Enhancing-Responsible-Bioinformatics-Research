Activity: Explore Reproducible MySQL Queries

Overview

This activity assumes a very basic understanding of the MySQL language. We will run simple queries on a small relational database hosted by the W3 schools.

If you would like to learn basic MySQL, our goal is not to reproduce excellent learning resources elsewhere here. Instead, try a formal class or you can visit the tutorials from the W3 schools:

  • MySQL Tutorial
  • SQL Tutorial
To begin this activity, go to the following page: MySQL TryIt Editor.

Once at the page, click the "Run SQL" button to run a query. A test query,

SELECT * FROM Customers;

should be pre-entered for you. Run that query to see what it returns. The actual contents of the database table is shown on the right of the page - examine the table to become familiar with. You can look at the attributes of each table using the DESCRIBE command:

DESCRIBE Customers;

Instructions

Consider you wanted to count the number products with a price over $55.00. What query would you use to do this? Write one and run it to see the answer your query provides.

Now, click the triangle to expand and try the queries below. What is the difference between these queries? Did you arrive at the same answer?



SELECT COUNT(Price) FROM Products WHERE Price > 55.00;

SELECT COUNT(Price) FROM Products WHERE Price >= 55.00;


Did you use COUNT()? Which answer is correct? (Or is the question too vague?)


What if you wanted to find how many products were available between $20 and $35. What query would you use to do this? Write one and run it to see the answer your query provides.

Now, click the triangle to expand and try the queries below. What is the difference between these queries? Did you arrive at the same answer?



SELECT COUNT(*) FROM Products WHERE Price BETWEEN 20 AND 35;

SELECT COUNT(*) FROM Products WHERE Price > 20 AND Price < 35;


Did you use COUNT()? Which answer is correct? (Or is the question too vague?)
Some tips for learning to write reproducible database queries are below:
  • Focus on understanding the actual question first and translating it into clear conditions before writing any code
  • Identifying which columns and filters are needed and then build your query one condition at a time, testing between each to confirm your query is doing what you anticipated.
  • Know the database contents and general attributes, elements, and tables well so you can catch when you may miss something.
  • Confirm a query is capturing the right data by testing your datasat using additional queries.
  • Know the syntax. Pay close attention to logical operators (in MySQL for example - AND vs OR and boundary differences like > vs >= or BETWEEN), since small changes can significantly affect results. Knowing the syntax and codebase well is supportive to this effort.
  • Explore the data with simple queries and compare similar queries to see how results differ.
  • If something looks wrong, revisit your logic rather than assuming the database is at fault, and aim for clear, readable queries instead of overly complex ones.
