Motivation

Curriculum design is more than choosing topics. A well-designed curriculum connects learning goals, instructional activities, and assessment so learners have repeated opportunities to practice, receive feedback, and build confidence. A curriculum should not only ask “What content should be covered?” It should also ask “What should learners be able to do, and how will they practice doing it?”. It also considers the cadence or pacing of learning, understanding that multiple touches on a topic over some period of time is better than "cramming" new information all at once.

This is especially important in computer science and bioinformatics because learners often need to integrate several kinds of knowledge at once: biological context, computational thinking, programming or tool use, statistics, data management, and scientific communication.

Outcomes

In the 1990's, the US higher education system shifted towards development of courses around student learning outcomes. Therefore, a good first step to developing any learning resource is to write learning objectives that describe what learners should be able to do. In computing education, competency-based curriculum frameworks are often used to connect knowledge areas, skills, and observable outcomes. The ACM/IEEE CS2023 curricular guidelines, for example, organize computer science education around knowledge areas and competency expectations rather than a simple list of topics.

Example

Outcome: Students will learn about databases.

(Better, Measurable) Outcomen: Students will compare flat-file, relational, and NoSQL database models and explain how database structure can affect reproducibility.

Alignment

Alignment means the learning objectives, practice activities, and assessments all point in the same direction.

Design element Question to ask Example
Learning objective What should learners be able to do? Choose an appropriate statistical test for a simple research scenario.
Practice activity How will learners practice that skill? Classify short scenarios by data type, number of groups, and test choice.
Assessment How will we know they can do it? A quiz or case study asking learners to justify test selection.
Check: If your objective says “analyze,” but your assessment only asks learners to define a term, the course is probably not aligned.

Inclusive Design

Learners enter bioinformatics with different backgrounds. Some may be strong in biology but new to computing. Others may be comfortable with code but less familiar with experimental design. Curriculum design should anticipate this variation.

Inclusive design practices

  • Define (or completely avoid) jargon at first mention
  • Provide examples, when appropriate, from multiple domains
  • Offer low-stakes practice before high-stakes assessment
  • Demonstrate use of clear file names, instructions, and expected outputs
  • Provide captions, transcripts, alt text, and keyboard-accessible materials
  • Help learners in a constructive way!

Beginner Learning Resource Design Checklist

  1. Write 1–2 measurable learning objectives
  2. Identify the background knowledge learners need before starting
  3. Break complex tasks into smaller steps
  4. Provide examples (I do, we do, you do)
  5. Include active practice, not just readings and lectures
  6. Give feedback that explains what to improve
  7. Include reflection so learners connect skills to their own work
  8. Check accessibility and clarity of instructions
  9. Revise based on learner performance and feedback
